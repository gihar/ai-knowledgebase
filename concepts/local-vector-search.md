# Локальный векторный поиск

Локальный векторный поиск — паттерн, при котором semantic/similarity search запускается рядом с приложением или прямо внутри его процесса, а не как отдельный managed-сервис. В контексте [[ai-engineering]] это снижает порог для RAG, персональных баз знаний, code search и memory-систем: разработчику не нужно сразу поднимать Pinecone, Weaviate, Qdrant или другой отдельный сервис.

## Почему это важно

Для небольших AI-продуктов и агентных workflow инфраструктура часто становится тяжелее самой задачи: нужен сервер, конфигурация, деплой, мониторинг, сетевой доступ и backup. Embedded-вариант вроде [[zvec]] меняет стартовую точку: установить пакет, создать коллекцию, положить embeddings в локальное хранилище и выполнять similarity/hybrid search из процесса приложения.

## Текущее состояние знаний

Пересланная заметка выделяет `zvec` как пример SQLite-подобного подхода к vector DB: база встраивается в процесс, хранится локально и не требует отдельного демона. Публичные материалы проекта подтверждают in-process дизайн, open-source статус, Python API, использование dense/sparse vectors, фильтров, full-text search и hybrid retrieval. Релиз `v0.5.0` добавил нативный полнотекстовый поиск, гибридный retrieval, DiskANN Index и Zvec Studio.

## Практические последствия

- Для прототипов RAG можно начинать с локального файла и embedded-библиотеки, а не с отдельной инфраструктуры.
- Для персональных knowledge base и локальных агентов embedded search упрощает переносимость и offline-режим.
- Hybrid search важен для прикладных баз знаний: keyword/full-text хорошо ловит точные термины, а vector search — смысловую близость.
- Disk-based индексы позволяют переносить часть нагрузки с RAM на диск, что важно для edge-устройств и больших локальных коллекций.
