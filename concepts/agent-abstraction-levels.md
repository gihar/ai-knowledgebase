# Уровни агентных систем

Уровни агентных систем — практическая шкала от простого чата с моделью до программно управляемого workflow, где код запускает много агентов, шагов и итераций. Эта шкала полезна для [[ai-engineering]], потому что помогает не путать маркетинговые названия фич с реальными источниками стоимости: контекстом, координацией, числом проверок и числом запусков модели.

## Шкала уровней

| Уровень | Что добавляет | Где появляется стоимость |
|---|---|---|
| Чат | Интерактивное взаимодействие пользователя с агентом и инструментами | Один основной контекст, действия и ответы в рамках диалога |
| Скилл | Переиспользуемая инструкция на естественном языке, иногда со скриптами | Обычно снижает стоимость: меньше повторного объяснения и меньше ошибок процедуры |
| Goal | Автономный цикл «план → действие → проверка» до критерия завершения | Растет число итераций, проверок и исправлений |
| Субагент | Отдельное окно контекста для подзадачи под управлением основного агента | Стоимость растет примерно по числу параллельных или последовательных подагентов |
| Agent team | Несколько ролей/агентов с обменом промежуточными результатами | Появляются токены на коммуникацию, ревью, согласование и конфликтующие гипотезы |
| Workflow | Код управляет агентами, шагами, ветвлениями, ретраями и критериями качества | Стоимость масштабируется как `агенты × шаги × итерации × ретраи` |

## Ключевое различие

Скилл — это не новый агент, а память о процедуре: markdown-инструкция, шаблон, чеклист или скрипт. Он делает поведение воспроизводимее и дешевле, но сам не создает автономный цикл.

Goal добавляет цикл выполнения до условия завершения. Субагент добавляет изоляцию контекста и параллелизм. Agent team добавляет многостороннюю координацию. Workflow переносит управляющую логику из «размышлений агента» в код, поэтому становится повторяемым, передаваемым и улучшаемым как software artifact.

## Почему стоимость растет нелинейно

Стоимость растет не потому, что следующие уровни «умнее» сами по себе, а потому что они запускают больше вычислительной работы вокруг LLM:

- больше отдельных контекстных окон;
- больше чтения и передачи промежуточных результатов;
- больше шагов plan/act/verify;
- больше ретраев и самопроверок;
- больше координации между ролями;
- больше логов, summaries и handoff-сообщений.

Поэтому подписи вроде `1 → N → N² → N×100` стоит читать не как точную формулу, а как интуицию: каждый следующий уровень покупает дополнительную организационную способность за токены, задержку и сложность отладки.

## Практические выводы

- Для повторяемой ручной процедуры сначала нужен скилл, а не команда агентов.
- Для задачи с ясным критерием завершения подходит goal-цикл.
- Для независимых подзадач полезны субагенты, особенно когда важна изоляция контекста.
- Для сложных задач с разными ролями нужен agent team, но только если коммуникационные накладные расходы оправданы.
- Для процесса, который нужно запускать много раз, передавать другим и улучшать, нужен workflow как код.
