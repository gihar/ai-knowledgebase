# Структурированное извлечение данных из документов

Структурированное извлечение данных из документов — workflow, где AI/OCR-система получает PDF или изображение и возвращает не просто текст, а валидированный объект: JSON, таблицу или набор полей по заранее заданной схеме. Для [[ai-engineering]] это важный мост между неструктурированными документами и downstream-системами: базами данных, Jira/ERP, аналитикой, compliance-проверками и агентными pipeline.

## Почему schema-first подход важен

Обычный OCR отвечает на вопрос «что написано на странице». Structured extraction отвечает на вопрос «какие поля нужны процессу и какими типами они должны быть». JSON Schema задаёт контракт: названия полей, вложенность, типы и обязательность. Это уменьшает постобработку и делает результат пригодным для автоматической проверки, сравнения с эталоном и загрузки в backend.

## Lift как пример

[[lift]] показывает современный open-source вариант этого паттерна: 9B vision-модель принимает PDF/images и JSON Schema, а затем возвращает JSON объект, соответствующий схеме. В пересланной заметке и публичном README важны три практических свойства: локальный/открытый запуск через `lift-pdf`, schema-constrained decoding и benchmark, где Lift близок к Gemini Flash 3.5 по field accuracy, но быстрее в заявленной конфигурации.

## Практические последствия

- Для внутренних документов можно строить pipeline без полного vendor lock-in на closed document AI API.
- Схема становится версионируемым артефактом: её можно хранить в Git, тестировать на fixtures и использовать в [[dynamic-workflows]].
- Агентные системы могут передавать извлечённый JSON дальше без чтения всей страницы в контекст LLM.
- Для критичных процессов нужна отдельная валидация: сравнение с эталонами, human review для low-confidence cases, контроль hallucinated/missing fields.

## Открытые вопросы

- Как Lift ведёт себя на русскоязычных документах, сканах плохого качества и табличных формах?
- Насколько хорошо schema-constrained decoding покрывает сложные JSON Schema-конструкции?
- Какой GPU/latency/cost профиль получится на реальных пакетах PDF, а не на benchmark examples?
